## 전반적인 내용과 강조 하고 싶은 내용

- spring data mongodb를 이용시 업데이트 작업에 대한 전반적인 가이드
- 크게 2가지 내용을 디름
- 특정 필드만 업데이트 하는 가이드
  - mongo db 같은 no sql 의경우 스카마가 매우 유연하며, 역정규화하는 것이 일반적이기 일반적이기 떄문에 이러한 영향으로 필드들이 많아 질수 밖에 없는 구조다.
  - 필드가 많아진다는 것은 단순히 저장할 데이터들이 많다는 것만은 의미하지 않는다. 해당 컬렉션이 다루는 컨텍스트 및 비지니스 항목들이 많아진다는 의미아다.
  - 이렇게 다양한 컨텍스트들이 혼재되어 있는 경우 모두 repository.save로 처리한다면 복잡한 컨텍스트 처리를 더 악화 시킨다. 수많은 필드 들이 어느 컨텍스트에 따라서 업데이트가    
    되는지 코드레벨로 관리하기가 어렵다. 사실상 도큐먼트 클레스의 save 모든 구간이 모든 필드가 업데이트가 가능한 구조다. 데이터 변경에 엄격한 도메인이라면 치명적인 단점이 될 수있다.
  - 예를 들어 가맹점 지급에 대한 수수료 정산 처리를 하는 도메인에서 의도하지 않게 수수료 필드의 변경을 야기할 수 있다. repository.save으로, 예를들어 동일한 PK에 대해서 A 사용자는 가맹점 이름을 A->B 으로 변경하고, 사용자 B는 수수료를 2% -> 3% 으로 변경한다고 가정해보자 이때 모두 save를 사용하게 되면, A사용자가 가맹점 이름을 변경하는 과정에서 B사용자가 수수료를 변경한 내용이 의도치 않게 덮어 씌워질 수 있다. 이런 문제를 방지하기 위해서는 특정 필드만 업데이트 하는 방법을 사용하고 명확하게 가르는 사용을 써야한다
  - 그래서 본 포스팅에서는 repository.save를 사용하지 않고, 특정 필드만 업데이트 하는 방법을 소개한다. 이때 spring data mongodb에서 제공하는 update 메서드를 사용한다. update 메서드는 특정 필드만 업데이트 할 수있도록 지원한다. 또 업데이트 관련된 필드들은 별도의 DTO클래스로 따로 만들고 해당 업데이트 하는 필드만 DTO에 담아서는 클래스를 한 곳에서 관리함으로써 어떤 필드가 어디에서 어떻게 업데이트 하는지 명확게 나타낸다.
- 업데이트 성능 가이드
  - 전반적인 성능 가이드는 Users/yun/yun/blog-sample/mongo-study/mongodb-bulk-update-performance.md 참고해서 만들것
  - 벌크 업데이트에 대한 전반적인 가이드, 이런 공통함수를 MongoCustomRepositorySupport 여기서 제공, jpa 비슷한 패턴으로 레포지토리를 확장할 수있다. 이런 내용 강조
  - 특별한 작업이 아니라 단순하게 i/o만 몰아서 처리하더라도 성능적인 이점이 크게 있다. 성능 개선도 병렬 처리, 멀티스레드 등등 직관적이지 않는 방법 보다는 이런 식을 먼저 처리를  
    해보고 추후에 더 성능ㅇ을 끌어 올릴 필요가 있다면 병렬철, 멀티 스ㅔ드 이런 방법을 고려하는 순서로 성능을 올리는것이 좋다. 처음 부터 너무 어렵고 직관적이지 않ㅇ는 방법으로 하면
    더 곤경에 처할 수있다.
