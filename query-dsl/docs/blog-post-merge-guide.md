# 여러 초안을 하나의 완결된 포스팅으로 통합하는 가이드

서로 다른 시점에 따로 작성된 초안(블로그 포스트, 기술 문서) 여러 개를 모순 없이 하나의 완결된 글로 합칠 때 쓰는 절차. `docs/querydsl-pagination-complete-guide.md`를 만들면서 실제로 이 절차를 밟았고, 8번(독립 검수) 단계에서 원본 문서에 수년간 잠복해 있던 컴파일 불가 버그 2건을 찾아냈다.

## 언제 쓰는가

- 통합 대상 문서가 2개 이상이고, 서로 다른 시점에 작성되어 연결성이 떨어질 때
- 문서가 실제 소스 코드를 인용하고 있어서, 시간이 지나며 문서와 코드가 어긋났을 가능성이 있을 때
- 단순 짜깁기가 아니라 하나의 서사로 읽히는 완결된 글이 목표일 때

## 입력

- 통합 대상 문서 2개 이상
- (있다면) 문서가 인용하는 실제 소스 코드

## 절차

1. **불일치 조사** — 문서 간, 그리고 문서 대 실제 소스 코드를 대조해 모순·중복·컴파일 안 되는 예제를 찾는다. 외부 CLI 자문(`omc ask`)이나 Explore 에이전트로 조사를 보강할 수 있지만, 지적된 내용은 반드시 직접 파일을 읽어 재확인한다. 자문이 틀릴 수 있다.
2. **사용자 결정** — 발견된 불일치마다 AskUserQuestion으로 하나씩 판단받는다. 절대 임의로 추측해서 진행하지 않는다. 이게 이 절차의 핵심 가치다 — 건너뛰면 그냥 자동 요약기가 된다.
3. **결정 로그 기록** — 모든 결정과 근거를 별도 마크다운 파일(예: `docs/<주제>-종합.md`)에 결정이 나오는 즉시 기록한다. "나중에 정리"하면 긴 세션 중간에 유실된다. 세션이 끊겨도 이 파일만 보면 복구 가능해야 한다.
4. **소스 코드 선(先) 수정** — 문서가 코드를 인용하므로, 결정된 버그/불일치는 문서보다 코드를 먼저 고친다. 동작 버그라면 회귀 테스트를 추가하고, 컴파일(가능하면 빌드·테스트 실행)로 검증한다.
5. **목차 스켈레톤** — 최종 문서의 헤딩 구조(H2/H3)를 먼저 확정하고, 각 섹션에 원본 출처(문서명 + 줄 번호) 또는 "재작성 대상"을 HTML 주석으로 매핑한다. 이 시점에 목차를 한 번 리뷰받는다.
6. **단계적으로 채우기** — 두 종류를 분리해서 진행한다.
   - **기계적 이식 구간**: 용어 통일(클래스명 등), 이미지 절대경로 통일, 이제 같은 글의 다른 섹션이 된 자기참조 링크를 내부 참조로 교체.
   - **완전 재작성 구간**: 결정으로 서사가 뒤집힌 부분(예: "정답"으로 소개하던 구현이 사실 버그였고 고친 버전을 보여줘야 하는 경우).
7. **브릿지 문단은 별도 패스로** — 원본 어디에도 없던 연결 문장(도입 훅, 장 전환, 복선 회수)은 6번과 섞지 않고 마지막에 독립된 단계로 쓴다. 섞으면 "새로 쓴 문장 vs 옮긴 문장"을 나중에 감사(audit)할 방법이 없어진다.
8. **독립 검수** — 작성한 본인이 검수하지 않는다. 별도 에이전트(verifier, code-reviewer 등 새 컨텍스트)에게 위임해 다음을 확인시킨다.
   - 인용한 코드 블록 ↔ 실제 소스 diff 대조
   - 이미지 링크 생존 확인 (원본 URL과 문자 단위 일치하는지)
   - 수치·용어 프레이밍 일관성 (같은 지표를 다른 방식으로 계산해 다른 곳에서 인용하고 있지 않은지)
9. **톤 다듬기** — 검수 반영 후 처음부터 끝까지 다시 읽으며 각 장의 도입 리듬이 일관되는지, 장 전환이 자연스러운지 확인한다.

## 성공 기준

- 코드펜스(```) 개수가 짝수 (렌더링 깨짐 없음)
- 작업용 플레이스홀더 주석이 0건
- 실제 소스를 인용하는 모든 코드 블록이 diff로 검증됨 ("맞겠지"가 아니라 직접 대조)
- 모든 비자명한 편집 결정에 추적 가능한 근거가 남아있음 (3번 결정 로그)
- 이미지 링크 개수가 원본과 일치

## 주의 — 실제로 부딪힌 함정

- **기계적 이식이 원본에 이미 있던 버그를 그대로 옮긴다.** "이식 구간이니 안전하다"고 넘기지 말 것. 실제 소스와의 diff는 이식 구간에도 똑같이 적용해야 한다. 이번 작업에서 컴파일 안 되는 코드 2건이 전부 "거의 그대로" 옮긴 구간에서 나왔다.
- **같은 통계를 다른 계산 방식으로 두 곳에서 인용하면, 우연히 비슷한 숫자가 나와도 같은 값이 아니다.** 프레이밍(예: "A 대비 B의 배수"인지 "A 자신의 처음 대비 끝 배수"인지)을 하나로 정하고, 문서 전체에서 다른 프레이밍이 남아있는지 grep으로 확인한다.
- **결정으로 바뀐 코드가 문서 여러 곳에 인용돼 있으면, 한 곳만 고치고 나머지를 놓치기 쉽다.** 변경된 메서드/클래스명을 인용하는 모든 코드 블록을 grep으로 찾아서 전부 갱신한다.
- **자기 검수는 놓친다.** 같은 컨텍스트에서 "다시 읽어봤는데 괜찮음"은 신뢰하지 말고, 반드시 새 컨텍스트의 에이전트에게 맡긴다.
- **원본 문서의 오타나 관례(예: 실제 파일명이 오타인 이미지 파일)를 "고치면" 링크가 깨질 수 있다.** 정정 대상과 유지 대상을 구분한다.
