## 사전 조사에서 드러난 문제

통합에 앞서 세 문서를 실제 소스와 대조한 결과, 문서 간 그리고 문서-소스 간 불일치가 확인됐다. 아래는 모두 소스를 직접 확인해 검증한 사실이다.

**1. 커서 문서가 실제 코드와 다르다.**
- `docs/querydsl-cursor-pagination.md`의 287줄, 334줄은 `applyCursorPagination`이 `cursorSelector: (T) -> String` 파라미터를 받는다고 서술한다. 그러나 실제 `Querydsl4RepositorySupport.kt:90-94`에도, 사용처인 `Payment4RepositoryImpl.kt:31-35`에도 그런 파라미터는 없다. **문서의 예제 코드는 현재 소스로 컴파일되지 않는다.**
- 문서 351줄은 `nextCursor: String?`(예시값 `"7"`)라고 하지만, 실제 코드 `Querydsl4RepositorySupport.kt:155`는 `nextCursor: T?`로 엔티티를 통째로 반환한다. "동작 원리" 절(224-282줄)의 예시값이 전부 실제 응답과 다르다.

**2. 커서 문서가 자기가 붙여넣은 코드와 모순된다.**
- 47줄은 "`pageSize + 1`개를 조회해서 `hasNext`를 판별한다"고 하는데, 바로 위 32-42줄에 붙여넣은 `applySlicePagination` 코드 자체가 `content.size >= pageable.pageSize`로 `+1` 조회를 하지 않는다. 실제 코드(`Querydsl4RepositorySupport.kt:63`, `QuerydslCustomRepositorySupport.kt:63`)도 마찬가지다.
- 반면 `paging-query-performance.md:126`은 같은 코드를 정확하게 서술하고 있다. **두 문서가 서로 반대되는 말을 한다.**
- 참고로 `applyCursorPagination`은 `limit(pageSize + 1)`(125줄) + `content.size > pageSize`(191줄)로 제대로 구현돼 있다. 즉 버그는 Slice 쪽에만 있다.

**3. 중복은 문서가 아니라 코드에 있다.**
- `QuerydslCustomRepositorySupport`(64줄)와 `Querydsl4RepositorySupport`(232줄)가 둘 다 실재하고, 후자가 전자를 완전히 포함하는 상위집합이다.
- `QuerydslCustomRepositorySupport`는 "죽은 코드"가 아니라 **실사용 중**이다: `Writer.kt:62`, `PaymentRepository.kt:18`, `OrderRepository.kt:55` 세 곳이 상속한다.
- 결정적으로 `applySlicePagination`의 **유일한 실제 호출처인 `OrderRepository.kt:142`가 이 버그 클래스를 상속**한다. `Querydsl4RepositorySupport`만 고치면 실행 경로의 버그는 그대로 남는다.

**4. 벤치마크 측정 환경이 원본 어디에도 없다.**
- MySQL 버전, 실행 환경, 측정 도구, 웜업 여부 중 어느 것도 세 문서에 기록돼 있지 않다. 지어낼 수 없다.

**5. `paging-query-performance.md:117-123` 표에 산술 오류가 있다.**
- "총 22개"라면서 표는 `5 + 5 + 5 + 5 + 3 = 23`이다. Page 4는 3이 아니라 2가 맞다.

## 3.1절 재작성안 (결정 2의 문서 측 대응)

원본은 `>=` 구현을 *정답*으로 소개하고, "끝까지 읽어봐야 안다"(B 127)를 *구조적 한계*로 설명한 뒤 B 131-137 표로 그 한계를 증명한다. `+1` 조회로 바꾸면 **그 표가 증명하는 것이 한계가 아니라 버그**가 된다. 따라서 버리지 않고 뒤집어서 재활용한다.

1. Slice 개념 (B 95 유지) — Count를 안 쓴다
2. **나이브한 구현과 함정** ← B 131-137 표를 *버그 재현*으로 재활용

   | 구현 | 총 22건, 요청 Size | 반환 | `hasNext` | 판정 |
   |------|-------------------|------|-----------|------|
   | `content.size >= pageSize` | 22 | 22 | **true** | ✗ 다음 페이지가 없는데 있다고 답함 |
   | `limit(size+1)` + `size > pageSize` | 22 | 22 | **false** | ✓ |

3. **해결: 한 건 더 읽는다** — `limit(pageSize + 1)`, `content.size > pageSize`
4. B 117-123 표 재작성 (22건, size 5 기준 — 결정 8 교정 반영)

   | Page | 요청 Size | 실제 조회 | 반환 Content | `hasNext` |
   |------|-----------|-----------|--------------|-----------|
   | 0~3 | 5 | 6 | 5 | true |
   | 4 | 5 | 2 | 2 | false |

5. B 127 "끝까지 읽어봐야 정확히 판단" → **"한 건만 더 읽어보면 알 수 있다"**로 교체
6. **복선**: "이 `+1` 기법은 7장 커서에서 그대로 다시 등장합니다"

코드 수정안 (결정 2). `.limit()`이 **`applyPagination` 뒤에 와야** 하고(덮어쓰기), 남는 1건을 `take`로 잘라내야 호출자에게 새지 않는다 — 원본 코드에 없는 디테일이다.

```kotlin
protected fun <T> applySlicePagination(
    pageable: Pageable,
    query: Function<JPAQueryFactory, JPAQuery<T>>
): Slice<T> {
    val jpaContentQuery = query.apply(queryFactory)
    val content = querydsl!!.applyPagination(pageable, jpaContentQuery)
        .limit((pageable.pageSize + 1).toLong())  // applyPagination이 건 limit을 덮어씀
        .fetch()
    val hasNext = content.size > pageable.pageSize
    return SliceImpl(content.take(pageable.pageSize), pageable, hasNext)
}
```

### 벤치마크 인용 3종은 성격이 달라 처리도 다르다

**결정 4는 셋 중 하나에만 적용된다.**

2. **C 443-508 limit/offset vs cursor** — ★ **결정 4 적용 대상**. SQL 블록의 `-- 소요 597 ms` 주석은 제거하고 구간 라벨만 남긴다. 표는 상대 배수로 전환:

   | 구간 | limit/offset | cursor |
   |------|--------------|--------|
   | 첫 번째 | 1.0배 (기준) | 1.0배 (기준) |
   | 약 50% | 2.0배 | 1.0배 |
   | 약 79% | 2.7배 | 1.0배 |
   | 마지막 | **3.3배** | 1.0배 |

   원본의 `+1.4% / +101% / +184% / +235.3%` 열은 배수와 퍼센트가 섞여 혼란스러우니 폐기한다. 원본의 "3.4배"는 *마지막 페이지에서 offset이 cursor보다*(1995/595 = 3.35)이고, 결정 4의 프레이밍인 *첫 페이지 대비 마지막 페이지*는 1995/597 = 3.34다. cursor가 평탄해서 두 값이 우연히 같으니 **"3.4배" 표현은 그대로 써도 무방**하되, 마무리 표(C 520-524)와 반드시 같은 프레이밍으로 통일한다.
3. **B 279/291 "1037ms"** — `Thread.sleep`으로 만든 인위적 지연(1000ms + 500ms 설계값)이라 측정 환경과 무관한 **동작 증명**이지 벤치마크가 아니다 → **유지**.

1장(`select`/`selectFrom`만) → 5장(페이징 메서드 **추가분만**) → 7장(커서 메서드**만**). 전문을 3번 반복하면 지루하니 매 지점은 증분만 싣고 마지막에 GitHub 전문 링크를 단다.
