## 사전 조사에서 드러난 문제

계획 단계(Planner → Architect → Critic)에서 실제 소스와 세 차례 대조 검증된 사실이다.

**1. 실제 소스가 이미 존재한다.**
`src/main/kotlin/com/example/querydsl/service/BatchInsertService.kt`가 Insert와 Update 메서드를 한 클래스에 이미 갖고 있다.
- `executeBulkInsertWritersWithSql(writers: List<Writer>): Long` — Insert. name/email/score/reputation/active 5개 필드 세팅.
- `executeBulkUpdateWritersWithSql(writers: List<WriterUpdate>): Long` — Update. `name` 필드 1개만 세팅 + `where(id.eq(id))` 필수.
- `updateWriters(writers: List<Writer>)` — dirty checking 비교군.
- 생성자: `BatchInsertService(jpaQueryFactory: JPAQueryFactory, jdbcTemplate: JdbcTemplate, writerRepository: WriterRepository, dataSource: DataSource)` — 4개 파라미터.
- `sqlQueryFactory`는 `by lazy` 공유 인스턴스(24-26줄).
- `data class WriterUpdate(val id: Long, val name: String)` — 필드 2개뿐.

**2. `querydsl-batch-update.md` 문서 안에서 같은 메서드가 서로 다른 버전으로 서술된다.**
- "구현 코드 예시"(39-70줄): `fun batchUpdate(writers: List<Writer>): Long` — 5개 필드 세팅.
- "성능 측정 코드"(158-185줄): `fun batchUpdate(writers: List<WriterUpdate>): Long` — 1개 필드(name)만 세팅. 동작은 실제 소스와 같지만 메서드명이 다름.
- 실제 소스: `executeBulkUpdateWritersWithSql` — 위 두 문서 버전 어느 쪽과도 이름이 다름.
- 생성자도 문서 내에서 0개(161줄 `BatchInsertService()`)와 1개(55-56줄 `dataSource`) 두 버전이 등장하며, 실제 소스는 4개.

**3. `querydsl-batch-insert.md`도 소스와 어긋난다 — "Insert는 안전"이 아니다.**
- 문서는 생성자 1개(`dataSource`)만 보여주지만 실제 소스는 4개.
- 문서는 `sqlQueryFactory`를 메서드 내부에서 생성하지만 실제 소스는 `by lazy` 공유 인스턴스.
- 단, 메서드명(`executeBulkInsertWritersWithSql`)과 필드 세팅 범위(5개)는 소스와 일치한다.

**4. 성능 수치는 서로 다른 연산·다른 기준선이다.**
- Insert: 10,000건 기준 96.5% 개선 (`saveAll`의 IDENTITY 전략 단건 Insert 대비)
- Update: 88.4%/88.2% 개선 (dirty checking N-update 대비)
- 100건 구간에서는 Update(89.3%)가 Insert(88.5%)보다 오히려 개선율이 높다 — 두 수치를 나란히 비교표로 놓으면 "Insert가 Update보다 우수하다"는 무의미하고 오도하는 결론을 유도한다. **배수/비교표 통일 금지.**

**5. 이미지**: 본문에서 실제 참조되는 이미지는 2개뿐이다 — `Insert-Performance.png`(insert.md 237줄), `Update_Performance.svg`(update.md 254줄). `before-update.png`, `ater-update.png`(오타), `update-query.png`, `insert-result.png`는 이 두 문서 어디서도 참조되지 않지만, `docs/jpa-bulk.md`·`docs/jpa-persistence-context.md`에서 이미 사용 중이므로 삭제 대상이 아니라 "이번 통합본에 삽입하지 않는다"는 의미로만 처리한다.

**6. 자기참조**: `querydsl-batch-update.md` 9줄이 Insert 문서의 외부 발행 URL(`cheese10yun.github.io/querydsl-batch-insert/`)을 "이전 포스팅"으로 링크한다. 통합되면 내부 참조로 바뀌어야 한다.

**7. 환경 제약**: 로컬 MySQL 접근 불가 이력이 있어(이전 페이징 통합 작업에서 확인), 실행 검증은 컴파일까지만 가능하다.
